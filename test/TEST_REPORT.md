## 3. 数据库 Schema

使用独立数据库 `region_coverer`，包含两张表：

### villages（村落主表）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL PK | 自增主键 |
| village_name | VARCHAR(100) | 区划名称 |
| province | VARCHAR(50) | 省份 |
| city | VARCHAR(50) | 城市 |
| county | VARCHAR(50) | 区县 |
| encrypted_geom | BYTEA | Fernet加密的WKB几何数据 |
| cell_count | INTEGER | S2覆盖Cell数量 |
| created_at | TIMESTAMP | 创建时间 |

索引：`idx_villages_county (province, city, county)`

### village_s2_cells（S2索引表）

| 列名 | 类型 | 说明 |
|------|------|------|
| cell_id | BIGINT | S2 Cell ID（Hilbert编码） |
| village_id | BIGINT FK | 关联村落ID |
| is_interior | BOOLEAN | true=内部Cell, false=边界Cell |
| level | SMALLINT | S2 level (12~18) |

索引：
- `PRIMARY KEY (cell_id, village_id)`
- `idx_vsc_cell (cell_id)`
- `idx_vsc_village (village_id)`
- `idx_vsc_level (level)`

## 9. 性能分析

### 查询性能

| 查询类型 | 平均耗时 | 典型范围 |
|----------|----------|----------|
| 点查询 | 12.6ms | 5~25ms |
| 面查询(小区域) | ~90ms | 80~100ms |
| 面查询(大区域) | ~500ms | 200~800ms |

### 入库性能

| 指标 | 数值 |
|------|------|
| 总入库时间 | 1,142s (~19min) |
| 平均每条入库 | 2.4s |
| 入库吞吐 | ~0.4 features/s |

### 性能瓶颈分析

1. **入库慢**：每个区划需生成 S2 覆盖（计算密集）+ 加密 geometry + 逐条写入DB
2. **面查询慢**：大面积查询生成的 S2 Cell 数量大，SQL 条件多，解密验证候选村落多
3. **Cell 数量过多**：L12 Cell 占 95%，大部分是粗粒度覆盖，索引膨胀

---

## 10. 优化建议

### 短期优化

1. **调高 max_cells**：从 500 提升至 2000~5000，减少 L12 粗粒度 Cell 占比
2. **降低 min_level**：从 12 降至 10，允许更大的内部 Cell
3. **批量入库优化**：使用事务批量 INSERT 而非逐条提交
4. **查询缓存**：对热点区域缓存查询结果

### 中期优化

1. **Cell 数量压缩**：对 L12 的 interior Cell 进行合并（用更少的 L10/L11 Cell 替代）
2. **分区索引**：按省份/城市对 village_s2_cells 做分区
3. **查询优化**：面查询时限制候选数，避免大面积查询返回过多结果

### 长期优化

1. **引入 PostGIS**：利用空间索引替代 S2 粗过滤，提升查询精度
2. **内存缓存层**：用 Redis 缓存热点区划的解密 geometry
3. **异步入库**：Kafka + 消费者模式，解耦入库与查询
